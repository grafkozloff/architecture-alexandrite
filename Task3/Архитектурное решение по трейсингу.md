## Мотивация
Внедрение трейсинга повысит управляемость и прозрачность бизнес-процессов компании:
* Быстрый ответ на вопрос "где мой заказ?": Вместо ручного расследования инженеры быстро увидят полный путь заказа по всем системам и точно определят, на каком этапе он застрял.
* Сокращение времени на диагностику сбоев: Вместо долгого сбора и сопоставления логов из разных систем инженер сразу видит не только то, где произошла ошибка, но и какой именно запрос её вызвал и в каком контексте.
* Объективные данные для оптимизации процессов: Трейсинг покажет, какие этапы (например, расчет стоимости или ожидание подтверждения в CRM) являются самыми медленными и создают "бутылочное горлышко".
## Предлагаемое решение
Для реализации трейсинга будет использован стек на основе OpenTelemetry (OTEL) и Jaeger.
### Новые компоненты:
* Jaeger Agent:  принимает данные от SDK и отправляет дальше
* Jaeger Collector:  принимает данные от Jaeger Agent, обрабатывает и пишет в БД
* Trace db (например, Cassandra): хранилище трейсов
* Jaeger Query: предоставляет API для доступа к данным
* Jaeger UI: веб-интерфейс для просмотра трейсов
### Доработки компонентов:
* Для следующих компонентов необходимо внедрить библиотеку OpenTelemetry SDK и настроить экспорт трассировок на Jaeger Agent:
  * Shop API
  * CRM API
  * MES API
  * Internet Shop
* Интеграция с RabbitMQ: Настроить сквозную передачу trace_id через заголовки сообщений.

## Компромиссы
Применение трейсинга не всегда возможно или целесообразно:
* Монолитные системы без распределенных вызовов. Если вся бизнес-логика выполняется внутри одного процесса (монолит), стандартные профилировщики и логирование справятся с задачей диагностики лучше и проще.
* Критичные к задержке системы реального времени. Любые накладные расходы на создание спанов, передачу заголовков и отправку данных в коллектор неприемлемы.
* Системы с гигантским трафиком. Стоимость хранения и индексации всех трейсов может стать запредельной.
* Проприетарными или legacy-системы. Система не может быть доработана или доработка будет слишком дорогой.

## Безопасность
Система трейсинга содержит чувствительные данные: полные цепочки выполнения бизнес-логики, идентификаторы заказов, имена сервисов и структуру коммуникации. Доступ к ней должен быть строго контролируемым.
* **Аутентификация:** Интеграция с системой аутентификации компании. Войти смогут только сотрудники с актуальной учётной записью.
* **Авторизация:** Настрорить RBAC: роль readonly для сотрудников поддержки и менеджеров, роль admin для инженеров и DevOps.
* **Защита внешнего доступа:** Доступ только через VPN с обязательной двухфакторной аутентификацией.
* **Защита каналов передачи данных:** Соединения должны использовать mTLS.
* **Сетевая изоляция:** Компоненты трейсинга должны быть развёрнуты в закрытом сегменте сети.
* **Политика хранения:** Настроить автоматическое удаление старых трейсов.
* **Аудит доступа:** Все попытки доступа к трейсам должны логироваться.

## Автоматический мониторинг процесса прохождения заказа
Цель: автоматически обнаруживать заказы, которые зависли или аномально долго проходят какой-либо этап, и уведомлять об этом команду.<br>
Новые компоненты:
* Tracing Monitor Service: Это кастомное приложение, которое:
  * Подписывается на поток спанов из Jaeger Collector.
  * Агрегирует спаны по trace_id и order_id.
  * Вычисляет длительность этапов и общее время выполнения заказа.
  * Сравнивает с пороговыми значениями.
  * При обнаружении отклонения генерирует событие (alert).
* Tracing Monitor db: Хранилище агрегиррованных состояний трейсов, key-value с TTL.
* Alert Manager (например, Prometheus Alertmanager): Принимает события от Tracing Monitor Service, создает уведомления (например, в Telegram) и тикеты на инциденты (например, в Jira).
