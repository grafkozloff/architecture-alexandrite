## Анализ
Нобходимо собирать логи, которые позволяют воспроизвести путь заказа, диагностировать ошибки и аудировать действия пользователей.
### Системы:
* Shop API (Backend): Логика создания заказа, корзины, взаимодействия с клиентом.
* CRM API (Backend): Логика работы менеджеров, смены статусов заказов.
* MES API (Backend): Логика расчета стоимости, управления производством.
* Message Queue (RabbitMQ): Логи доставки сообщений (опционально, для диагностики сложных проблем).
* Базы данных (PostgreSQL): Ошибки, медленные запросы.
### Список логов уровня INFO:
| **Система**      | **Событие**                              | **Данные**                                                                       |
|------------------|------------------------------------------|----------------------------------------------------------------------------------|
| **Все сервисы**  | Вход пользователя в систему              | Событие, время, идентификатор пользователя, роль                                 |
| **Shop API**     | Создание заказа                          | Событие, время, идентификатор покупателя, номер заказа                           |

### Другие уровни логирования
* **ERROR:** Операция не может быть выполнена из-за сбоя (исключение, недоступность сервиса, ошибка валидации ии т.д.).
* **WARN:** Операция выполнена, но произошло что-то нестандартное, что может указывать на потенциальную проблему (например, слишком долгий расчет стоимости).
* **DEBUG:** Детальная информация, полезная только разработчикам во время отладки (например, тело запроса/ответа от внешнего API, промежуточные шаги сложного алгоритма).
# Мотивация
Внедрение централизованного сбора структурированных логов обеспечит:
* **Детальную диагностику:** Трейсинг показывает "что сломалось и где", а логи отвечают на вопрос «почему это произошло». В логах содержится детальная информация об ошибках (стектрейсы, входные параметры), которую невозможно поместить в трейс.
* **Аудит действий пользователей:** Логи позволяют становить, «кто, что и когда сделал» — какой менеджер подтвердил заказ, какой оператор его запустил в производство.
* **Восстановление полной картины события:** При сложном инциденте совмещение трейса и логов позволяет восстановить хронологию и причину сбоя.
### Метрики
| **Метрика**                          | **Описание влияния**                                                                                                                                         |
|--------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Mean Time To Resolve (MTTR)**      | **Сократится.** Возможность instantly найти детали ошибки (стектрейс, параметры запроса) в едином интерфейсе ускорит фиксацию багов.                         |
| **Error Rate (%)**                   | **Деталиизируется.** Логирование позволяет точно считать не только факт ошибки (5xx), но и классифицировать их по типам (например, error="payment_failed").  |
| **Operator Efficiency**              | **Появится.** Логи действий операторов в MES покажут, сколько заказов они обрабатывают, на каких этапах возникают задержки.                                  |
| **Customer Dispute Resolution Time** | **Сократится.** Возможность быстро найти в логах, кто и когда менял статус заказа, ускорит решение споров.                                                   |
### Приоритизация логирования и трейсинга
Наиболее приоритетные системы для внедрения логировапия и трейсинга:
* **MES API:** Это самое узкое и проблемное место. Сервис рассчитывает стоимость, меняет статусы заказа и управляет производством. Логирование и трейсинг покажут, почему расчеты зависают, где и почему теряются заказы.
* **CRM API:** Через него проходят решения менеджеров. Логирование необходимо для аудита. Он также участвует в смене статусов.
## Предлагаемое решение
### Архитектура
Для реализации централизованного логирования будет развернут стек на основе ELK.
#### Новые компоненты:
* **Filebeat:** Сбор и пересылка логов.
* **Logstash:** Обработка и обогащение логов.
* **Elasticsearch:** Хранение и индексация логов.
* **Kibana:** Визуализация и анализ логов.
#### Доработки компонентов:
* **Все сервисы (Shop API, CRM API, MES API):** Настроить вывод логов в структурированном формате (JSON) в stdout.
* **Все инстансы (EC2):** Установить и настроить Filebeat в качестве демона.

### Безопастность
1. **Классификация данных и очистка логов:** Логи не должны становиться каналом утечки чувствительных данных. Необходимо исключить запись следующих данных:
   * Персональные данные: имена клиентов и сотрудников, телефоны, почта, адреса, паспортные данные, номера карт и т.д.
   * Системные данные: пароли, ключи, токены, системные аккаунты, внутренние IP-адреса и т.д.

   На уровне приложения необходимо либо не логировать чувствительные поля совсем, либо маскировать значения. Дополнительно, на уровне Logstash можно настроить фильтры для поиска и замены (или удаления) чувствительных данных.
2. **Контроль доступа к логам:** Доступ к логам должен быть строго регламентирован по принципу least-privilege:
   * Ролевая модель доступа в Kibana: Роль "Просмотрщик логов" для разработчиков, QA и Продакт-менеджера, роль "Администратор логов" для DevOps.
   * Интеграция с единой системой аутентификации: Доступ к Kibana должен быть предоставлен только сотрудникам компании через VPN из защищенной сети.
   * Доступ к бэкенду: Прямой доступ к кластеру Elasticsearch должен быть запрещен для всех, кроме инфраструктурной команды, и осуществляться через VPN из защищенной сети.
3. **Шифрование данных:**
   * Передача данных: Все соединения между компонентами стека должны использовать TLS.
   * Хранение данных: Диски, используемые Elasticsearch, должны быть зашифрованы.
4. **Управление жизненным циклом логов (Retention Policy):** Необходимо настроить в Elasticsearch сроки хранения логов в зависимости от категории.
5. **Мониторинг и аудит доступа к логам:** Необходимо фиксировать все попытки доступа к логам в Kibana и Elasticsearch.
### Политика хранения
Политика определяет правила хранения, ротации и удаления логов для баланса между стоимостью хранения, производительностью и потребностями бизнеса.
### 1. Стратегия индексов: Один индекс на систему в день
Имена индексов будут иметь шаблон: {service-name}-{yyyy-MM-dd}.
* Упрощение управления: Легко определить, логи какой системы занимают много места.
* Гибкость настроек: Можно назначить разные политики хранения для разных систем (например, логи MES API хранить дольше, чем логи CRM API).
* Производительность: Поиск и агрегация выполняются быстрее в рамках одного, более узкого индекса.
* Безопасность: Проще ограничить доступ на уровне индексов (например, право "Просмотр логов" только на индексы mes-api-*).
### 2. Политика хранения (Retention Policy)
| **Система / Назначение логов**                 | **Срок хранения** | **Примерный расчет размера (в день)**                | **Hot Phase (SSD)** | **Warm Phase (HDD)** |
|------------------------------------------------| --- |------------------------------------------------------| --- | --- |
| **MES API**  <br>(Ядро бизнеса, ошибки, аудит) | **90 дней** | Высокая нагрузка, детальные логи.  <br>**~10-15 ГБ** | 7 дней | 83 дня |
| **CRM API**  <br>(Аудит действий менеджеров)   | **60 дней** | Средняя нагрузка.  <br>**~2-5 ГБ**                   | 7 дней | 53 дня |
| **Shop API**  <br>(Диагностика ошибок)         | **30 дней** | Высокий трафик, но менее критичны.  <br>**~5-10 ГБ** | 7 дней | 23 дня |
| **Технические и служебные логи**  <br>(Ошибки) | **30 дней** | Для диагностики.  <br>**~2-5 ГБ**                    | 7 дней | 23 дня |
Общий объем хранилища (примерно без учета реплик): ~20-35 ГБ в день * 90 дней = ~1.8-3.2 ТБ.
### 3. Фазы жизненного цикла индекса
Для каждого индекса будет настроена политика ILM с тремя основными фазами:
1. **Hot Phase (Горячая фаза, 7 дней):**
   * Для оперативной работы с самыми свежими логами.
   * Хранение на быстрых SSD-дисках с репликацией.
   * Индекс остается доступным для частой записи и чтения.
2. **Warm Phase (Теплая фаза, до истечения срока хранения):**
   * Для исторического анализа и аудита.
   * Хранение на более медленных и дешевых HDD-дисках без репликации.
   * Индекс становится read-only, что ускоряет выполнение запросов только на чтение.
3. **Delete Phase (Фаза удаления):**
   * По истечении срока хранения (90, 60, 30 дней) индекс автоматически удаляется.
### 4. Критерии и исключения
* Важные инциденты: Логи, связанные с расследованием критичных инцидентов, могут быть помечены тегом (например, retention_extension: true) и их жизнь будет продлеваться вручную.
* Юридические требования: Если появятся требования регуляторов хранить логи аудита (логины, смены статусов) дольше, для индексов будет создана отдельная политика с перемещением в Cold Phase в объектное хранилище (например, в S3) с помощью плагина Elasticsearch Snapshot.
## Система анализа логов
### 1. Настройка проактивного алертинга на основе логов
| **Тип алерта**                | **Что отслеживает**                                                                 | **Почему это важно**                                                                        | **Как реализовать (пример)**                                             |
|-------------------------------|-------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------|--------------------------------------------------------------------------|
| **Резкий рост ошибок**        | Rate ошибок (5xx, 4xx) на ключевых сервисах (MES API, Shop API).                    | Обнаруживает сбои в работе системы до того, как они станут массовыми.                       | Правило: when rate() of logs with level:ERROR over 5min > 10.            |
| **Падение бизнес-активности** | Резкое снижение числа успешных событий "Создание заказа" или "Стоимость рассчитана" | Может указывать на "тихий" сбой, когда ошибки не логируются, но бизнес-процесс остановился. | Правило: when count() of event:order_created over 10min < 5.             |
| **Подозрительная активность** | Множественные неудачные попытки входа в CRM/MES с одного IP.                        | Обнаружение брутфорс-атак или попыток несанкционированного доступа.                         | Правило: when unique count() of failed_login from one IP over 2min > 20. |
| **Аномалия в данных**         | Появление новых, нестандартных значений в поле error или status.                    | Помогает обнаруживать неизвестные ранее проблемы или изменения в поведении системы.         | Использовать machine learning в ELK или сторонних инструментах.          |
### 2. Поиск аномалий и выявление угроз
* **Обнаружение DDoS-атак и скачков трафика:** Резкий рост числа запросов к публичным эндпоинтам Shop API с одного или множества IP-адресов. Необходимо ннастроить алерт, который автоматически подаст сигнал для блокировки атакующих IP-адресов.  
* **Обнаружение аномалий с помощью Machine Learning:**
  * ELK: Встроенный модуль Machine Learning может обучиться на исторических данных для обнаружения аномалий.
  * Сторонние системы: Можно экспортировать логи в специализированные SIEM-системы (например, Splunk) или платформы для анализа безопасности, которые имеют более продвинутые корреляционные правила.
### 3. Создание операционных и бизнес-дашбордов в Kibana
* Операционный дашборд (для DevOps):
  * Скорость поступления логов по сервисам, топ ошибок, скорость обработки сообщений в очередях (если логируются).
* Бизнес-дашборд (для Продакт-менеджера и руководителей):
  * Метрики: Количество созданных заказов в час, конверсия (успешные заказы / ошибки оформления), среднее время расчета стоимости.
  * Визуализации: Геокарта заказов, самые популярные изделия.
### 4. Интеграция с другими системами
* С трейсингом (Jaeger): Связывать ошибки в логах с конкретным trace_id для мгновенного перехода к полному контексту выполнения запроса.
* С метриками (Prometheus): Коррелировать всплески ошибок в логах с метриками нагрузки на сервисы (CPU, memory).

* Grafana Loki: Подходит для быстрого старта. Позволяет быстро получить работающую систему с малыми затратами и сложностью.
* ELK / OpenSearch: Возможно избыточено. Подходит, если есть требования к сложному анализу и высокому уровню безопасности.
