## Мотивация
Анализ описания компании содержит описание проблем: просроченные заказе, потерянные контракте и т.д. Одной из причин этих проблем является отсутствие мониторинга.
Добавление мониторинга позволит решить следующие вопросы:
### 1. Ускорение диагностики инцидентов и проблем
* **Визуализация потока заказа:** Можно будет в реальном времени увидеть, на каком этапе и в какой системе "застревают" заказы. Например, дашборд покажет: "1000 заказов в статусе SUBMITTED (ждут расчета цены в MES), 500 — в статусе PRICE_CALCULATED (ждут подтверждения в CRM)".
* **Определение узких мест:** Можно будет в увидеть, что, ннапример, MES обрабатывает 5 расчетов в минуту, а в очередь приходит 10 — отсюда и рост задержек.
### 2. Проактивное обнаружение проблем вместо реактивного разрешения инцидентов
* **Система раннего предупреждения:** Можно настроить алерты (оповещения) до того, как ситуация станет критической.
### 3. Обоснование архитектурных решений и инвестиций в инфраструктуру
* **Данные для принятия решений:** Графики загрузки CPU и памяти сервисов, дискового I/O и нагрузки на базы данных наглядно покажут, какие компоненты являются узкими местами.
* **Измерение эффективности изменений:** После внедрения кеширования или перехода на асинхронный API можно будет проверить их эффективность через снижение нагрузки на БД и уменьшение времени отклика.
### 4. Повышение Надежности и Предсказуемости Бизнес-Процессов
* **Бизнес-метрики:** Можно будет отслеживать: 
  * Количество новых заказов в час/день.
  * Среднее время выполнения заказа (от SUBMITTED до SHIPPED).
  * Процент просроченных заказов.
* **Планирование мощностей:** Понимание текущей и прогнозируемой нагрузки позволит бизнесу принимать взвешенные решения о расширении производства или, наоборот, о приостановке приема новых заказов при перегрузке.
## Выбор подхода к мониторингу
Предлагается использовать гибридную стратегию мониторинга:
* "Четыре золотых сигнала" для бизнес-логики
* USE для инфраструктуры

"Четыре золотых сигнала" идеально подходят для мониторинга именно внешнего поведения системы с точки зрения пользователя.

| **Сигнал** | **Что мониторить (пример)**                                                                                                                                                                  |
| --- |----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| ****Задержка (Latency)**** | • MES API: Время расчета стоимости.  <br>• Shop API: Время ответа на запросы каталога.  <br>• CRM API: Время загрузки списка заказов.                                                  |
| ****Трафик (Traffic)**** | • Shop API: RPS (requests per second) на эндпоинты оформления заказа.  <br>• Message Queue: Rate публикации/потребления сообщений.                                                       |
| ****Ошибки (Errors)**** | • Все API: Rate HTTP-ответов 5xx и 4xx.  <br>• Message Queue: Rate сообщений, отправляемых в DLQ (Dead Letter Queue).  <br>• Процессоры сообщений: Rate неудачных обработок сообщений. |
| ****Насыщение (Saturation)**** | • Message Queue: Длина очереди (например, price_calculation).  <br>• MES API: Загрузка CPU и Memory.  <br>• MES db: Нагрузка на диск и CPU.                                          |

"Четыре золотых сигнала" показывают что есть проблема, но не всегда точно указывают почему. Метод USE лучше подходит для глубокого диагностического анализа конкретных ресурсов, когда мы уже знаем, что проблема есть.

| **Ресурс** | **Что мониторить (пример)**                                                                                                                                                                                         |
| --- |---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| ****Серверы (EC2)**** | • ****Utilization:**** Загрузка CPU и Memory %, Использование диска %.  <br>• ****Saturation:**** Время ожидания CPU (Steal Time) %, Длина очереди запросов к диску. <br>• ****Errors:**** Потерри сетевых пакетов. |
| ****Базы данных**** | • ****Utilization:**** Количество используемых подключений к базе данных / макс.  <br>• ****Saturation:**** Длина очереди запросов к диску.  <br>• ****Errors:**** Неудачные запросы, дедлоки.                     |
| ****Очередь (RabbitMQ)**** | • ****Utilization:**** Дисковое пространство, используемое для очередей.  <br>• ****Saturation:****  Длина очереди. <br>• ****Errors:**** Сбои подтверждений (ACK), Ошибки публикации сообщений.           |

## Метрики
### 1. Message Queue (RabbitMQ)

| **Метрика**                                     | **Назначение**                                                                                                                                      | **Ярлыки**      |
|-------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------|-----------------|
| ****Number of messages in queue****             | Показывает насыщение. Рост очереди — прямой признак, что MES не справляется с нагрузкой.                                                            | queue           |
| ****Number of message in flight in RabbitMQ**** | Показывает, сколько сообщений взято потребителями, но еще не обработано (не ack'нуто). Высокое значение может указывать на "зависшие" задачи в MES. | queue, consumer |
| ****Number of dead-letter-exchange letters****  | Сообщения, которые были отброшены после множества неудачных попыток обработки. Потеря бизнес-событий (заказов).                                     | queue, reason   |

| ****Message deliver rate****                    | Показывает скорость потребления сообщений MES. Сравнение с publish rate показывает, отстает ли потребитель.                                         | queue, consumer |

## План действий
### Развертывание стека сбора и визуализации метрик
* Развернуть Prometheus для сбора метрик и Alertmanager для оповещений.
* Развернуть Grafana для визуализации. Создать дашборды.
* Настроить алерты в на критические события.
###  Интеграция мониторинга ключевых компонентов
* Сервисы (Java/C#): Настроить экспорт метрик в Prometheus.
* RabbitMQ: Включить и настроить встроенный Prometheus-плагин для сбора метрик по очередям.
* Базы данных (PostgreSQL): Настроить стандартный postgres_exporter для сбора базовых метрик.
* Серверы (EC2): Установить Node Exporter для сбора метрик.

### Действия при превышении порогов
1. Для порогов **Warning** уровня:
   * Система автоматически создает тикет в трекере задач (например, Jira) и отправляет уведомление команде разработки и DevOps. 
2. Для порогов **Critical** уровня:
   * Срабатывает "звонилка" — автоматический звонок ответственным инженерам.
   * Отправляется срочное сообщение в канал инцидентов с высоким приоритетом.
   * Система пытается автоматически стабилизировать ситуацию (если это настроено):
     * Длина очереди RabbitMQ > 1000: Система мониторинга автоматически отправляет команду в AWS Autoscaling Group на увеличение количества инстансов сервиса потребителя, чтобы быстрее обрабатывать очередь.
     * Число подключений к БД > 90%: Автоматически запускается скрипт, который пытается найти и "прибить" долгие idle-сессии или запросы.
     * Свободное место на диске < 10%: Автоматически запускаются скрипты очистки: удаляются старые логи, временные файлы, кеши.
