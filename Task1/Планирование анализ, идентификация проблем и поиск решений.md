## Существующие и потенциальные проблемные места
### 1. Проблемы с производительностью и масштабируемостью
* Долгая загрузка дашборда в MES: Первая страница MES, которая является критически важной для операторов, долго грузится. Пагинация и фильтрация не решили проблему.
* Долгий расчет стоимости в MES: Расчет стоимости изделия может занимать до 30. При это для партнеров открыт синхронный API непосредстивенно в MES. При поступлении массовых синхронных вызовов проблема с проризводительностью усугубляется.
* Отстутсвие масштабируемости: В текущей архитектуре имеется по одному инстансу каждого приложения. Базы данных используются одновременно на чтение и запись. При росте нагрузки текущая архитектура не справляется. Базы данных и сервисы (особенно MES) становятся точками отказа.
* Отсутствие кеширования: Нет кеширования ни на каком уровне (база данных, уровень приложения, уровень веб-сервера). Это приводит к избыточной нагрузке на сервисы и базы данных.
### 2. Проблемы надежности и отказоустойчивости
* Отсутствие защиты от каскадных сбоев и перегрузки: Интеграция CRM API -> MES API через RabbitMQ не защищена от сбоев и перегрузки. Если MES перегружена и не может обрабатывать сообщения из очереди, это может привести к переполнению очереди и потере сообщений о заказах или сбое в сервисе CRM.
* Синхронные вызовы для долгих операций: Прием заказов от других продавцов инициируется синхронно через MES API, выполняющей длительныее рассчеты. Это приводит к блокировке ресурсов, перегрузке и, как следствие к таймаутам и сбоям.
* Единая точка отказа: Каждый сервис и каждая база данных развернуты в единственном экземпляре. Падение любого узла приводит к простою.
### 3. Проблемы отслеживаемости (Observability)
* Отсутствие мониторинга, логирования и трейсинга: Нет никакой информации о системе мониторинга, логирования и трейсинга. Это является причиной, почему команда не может:
* Быстро определить причину медленной загрузки дашборда MES.
* Понять, где именно копятся заказы и в каком статусе они "зависают".
* Видеть нагрузку на каждый сервис и предсказывать проблемы до того, как клиенты начнут жаловаться.
* Анализировать производительность интеграций между сервисами.
### 4. Проблемы процессов и эксплуатации
* Ручное тестирование: Процесс QA основан на ручном тестировании, что является медленным и ненадежным. Единственный QA инженер справляется все хуже и хуже, что приводит к задержкам релизов.
* Ручной деплой на release и prod: Отсутствие автоматизированного пайплайна для staging и production-окружений увеличивает риск человеческой ошибки при развертывании и замедляет поставку изменений.
* Нехватка экспертизы в команде: В команде только один разработчик на C# (поддерживает MES, которая стала критическим узким местом) и один DevOps-инженер на всю растущую инфраструктуру.
## Инициативы для устранения нежелательных ситуаций
### 1. Внедрение асинхронного API для внешних продавцов
**Описание:** Замена синхронного вызова MES API на асинхронную схему. Запрос будет помещаться в очередь. Результат расчета будет возвращаться через другую очередь или webhook.<br/>
**Решаемые проблемы:** Долгий расчет стоимости, блокировка ресурсов, перегрузка MES из-за синхронных вызовов.
### 2. Внедрение кеширования на базе Radis
**Описание:** Применить паттерн кэширования Cache-Aside для сервисов Shop API и MES API. Для Shop API кешировать каталог и информацию о товарах. Для MES API кешировать информацию о заказах для построения дашбода и детали конкретных заказов с коротким TTL и инвалидацией кеша по ключу при обновлнии заказа.<br/>
**Решаемые проблемы:** Долгая загрузка дашборда MES, избыточная нагрузка на базу данных. С онлайн-магазином проблем вроде бы нет, но можно улучшить пользовательский опыт.
### 3. Повышение масштабируемости и отказоустойчивости сервисов
**Описание:** Настроить EC2 Auto Scaling для сервисов и балансировщики нагрузки между инстансами.<br/>
**Решаемые проблемы:** Отсутствие масштабирумости и низкая отказоустойчивость.
### 4. Репликация баз данных
**Описание:** Для баз данных (MES db и Shop DB) настроить репликацию.<br/>
**Решаемые проблемы:** Единая точка отказа.
### 5. Репликация RabbitMQ
**Описание:** Настроить репликацию RabbitMQ.<br/>
**Решаемые проблемы:** Единая точка отказа.
### 6. Реализация backpressure для интеграции CRM API -> MES API через RabbitMQ 
**Описание:** Настроить prefetch_count на стороне MES и мониторинг очереди на стороне CRM для предотвращения "заваливания" MES API новыми сообщениями и переполения очереди необработанными сообщениями.<br/>
**Решаемые проблемы:** Отсутствие защиты от каскадных сбоев и перегрузки.
### 7. Внедрение мониторинга, логирования и трейсинга
**Описание:** Внедрить сбор метрик (CPU, память, время ответа API, запросы, ошибки) со всех сервисов и баз данных с помощью Prometheus. Настроить дашборды в Grafana. Настроить централизованный сбор и анализ логов с помощью ELK. Внедрить трейсинг с помощью Jaeger.<br/>
**Решаемые проблемы:** Отсутствие мониторинга, невозможность диагностики.
### 8. Автоматизация тестирования
**Описание:** Выделить время на создание набора автоматизированных тестов. Интегрировать их в CI/CD пайплайн.<br/>
**Решаемые проблемы:** Ручное тестирование, задержки релизов.
### 9 Автоматизация деплоя в release и prod окружения
**Описание:** Настроить пайплайн CI/CD. Это ускорит поставку изменений и снизит риск человеческой ошибки.<br/>
**Решаемые проблемы:** Ручной деплой, медленная поставка изменений.
### 10. Найм и обучение команды
**Описание:** Обосновать и инициировать найм: 1) второго разработчика C# для поддержки MES; 2) инженера по автоматизации тестирования (QA Automation) для разгрузки единственного QA; 3) SRE-инженера или Performance-инженера.<br/>
**Решаемые проблемы:** Нехватка экспертизы, перегрузка ключевых специалистов.
## Приоритизированный список инициатив
### Волна 1: Немедленные действия (квартал 1)
1. Внедрение асинхронного API для внешних продавцов.<br/>
   Синхронные вызовы напрямую в MES блокируют потоки, вызывают таймауты и не дают системе обрабатывать нагрузку. Переход на очереди мгновенно разгрузит MES, устранит таймауты для партнеров и позволит системе обрабатывать заказы в своем темпе.
2. Внедрение мониторинга, логирования и трейсинга.<br/>
   Без мониторинга мы не увидим, как ведет себя систма после изменений, и где новые узкие места. Логирование и трейсинг помогут разобраться в инцидентах. 
3. Реализация backpressure для интеграции CRM API -> MES API через RabbitMQ.<br/>
   Важно предтвратить потерю сообщений и разгрузить MES API.
### Волна 2: Стабилизация (квартал 2)
4. Репликация RabbitMQ.<br/>
   Очередь является критичным компонентом, её падение парализует всё производство. Репликация устраняет единую точку отказа и делает messaging-инфраструктуру отказоустойчивой.
5. Внедрение кеширования на базе Redis.<br/>
   Самая болезненная проблема для операторов — тормозящий дашборд. Кеширование списков заказов в MES API улучшит работу операторов и снизит нагрузку на MES db. Кеш для каталога Shop API тоже важен, но менее критичен прямо сейчас.
6. Репликация баз данных.<br/>
   Базы данных — это следующие по критичности точки отказа.
### Волна 2: Развитие (квартал 3+)
7. Повышение масштабируемости и отказоустойчивости сервисов (Auto Scaling).<br/>
   Это дорого, при этом репликация БД и очереди уже существенно улучшают ситуацию.
8. Автоматизация деплоя в release и prod окружения.<br/>
   Не влияет на систему непосредственно, но позволяет быстро и безопасно выпускать исправления и улучшения.
9. Автоматизация тестирования.<br/>
   Аналогично предыдущему пункту.
10. Найм и обучение команды.<br/>
    Это не техническая, а управленческая инициатива, которую нужно запускать параллельно с самого начала. Без усиления команды реализация всего плана затянется.
### Архитектура через полгода:
* **Асинхронная обработка:** Все долгие операции (расчет стоимости) работают через очереди. Партнеры и клиенты не испытывают таймаутов.
* **Защита от сбоев:** Критические компоненты (RabbitMQ, БД) реплицированы и не являются едиными точками отказа. Механизмы backpressure позволяют не терять сообщения о заказах и не дают MES "утонуть" в задачах.
* **Полная наблюдаемость:** Команда имеет единый дашборд в Grafana, где видна нагрузка на все сервисы, длина очередей и бизнес-метрики. Логи доступны для поиска.
* **Производительность:** Дашборд MES закеширован и грузится мгновенно. Операторы довольны.
### Три самых приоритетнных пункта:
1. Внедрение асинхронного API для внешних продавцов.
2. Внедрение мониторинга, логирования и трейсинга.
3. Реализация backpressure для интеграции CRM API -> MES API через RabbitMQ.

Эти пункты решают корневые проблемы бизнеса и создают фундамент для дальнейших улучшений.
